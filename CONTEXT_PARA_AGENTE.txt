================================================================================
SISTEMA DE ALIASES — NOMES DIFERENTES DO MESMO CLIENTE/PESSOA
================================================================================

PROBLEMA
--------
O SAP registra nomes de clientes que podem ser diferentes dos nomes usados no
budget (ex: "BANCO BMG S.A." no SAP vs "BMG" no budget). Antes da correção,
isso criava linhas duplicadas em branco na aba Clientes do dashboard.

SOLUÇÃO — campo nome_base em parametros.xlsx/clientes
------------------------------------------------------
Cada linha em "clientes" representa um cliente canônico com:
  - nome_cliente: nome exibido no dashboard (nome oficial)
  - nome_base:    nome(s) como aparecem no SAP, separados por | quando múltiplos
  - bu:           vertical/BU (ex: "Grupo Mult", "Health", "Finance")
  - ae:           nome do AE responsável

Exemplos:
  nome_cliente="SADA"    nome_base="SADA TRANSPORTES|SADA TRANSPORTES E ARMAZENAGENS LTD"
  nome_cliente="C6 BANK" nome_base="BANCO C6 S.A."

COMPORTAMENTO NO CÓDIGO
-----------------------
- main.py: read_clientes_csv() lê parametros.xlsx/clientes
           write_clientes_csv() salva de volta em parametros.xlsx/clientes
           (endpoint /api/clientes/ae escreve nessa aba)
- apuracao_engine.py: usa nome_base para normalizar nomes SAP antes dos lookups
  → _alias_pre[norm(nome_base)] = norm(nome_cliente)  — SAP → canônico
  → aliases adicionados nos lookups de receita/margem/custo após o groupby
- main.py get_clientes_list(): constrói alias_to_canonical map e agrega receita
  de TODOS os aliases em uma única linha na aba Clientes

REGRA IMPORTANTE
----------------
Nunca criar nova linha em clientes para um alias do mesmo cliente.
Sempre adicionar o alias no campo nome_base, separado por |.
Exceção: KLABIN aparece em 2 linhas porque tem BU diferente (Multisector + Grupo Mult).

ERROS CORRIGIDOS RELACIONADOS
------------------------------
- BMG tinha 3 linhas (1 principal + 2 aliases GRUPO BMG/Grupo BMG obsoletos).
  Após projetos.csv ser migrado para usar "BANCO BMG S.A.", os aliases ficaram
  sem correspondência. Resultado: 2 linhas em branco na aba Clientes, a linha
  com dados caía na posição ~50 (limite da página 1), usuário via só os em branco.
  FIX: removidas as 2 linhas alias obsoletas (commit 1a04e81).

================================================================================
REGRAS DE NEGÓCIO — BÔNUS Q4 2025
================================================================================

POSIÇÕES
--------
- DIRETOR: TCV(20%) + Receita(30%) + MB%(10%) + MC%(40%) — calc_bonus_diretor()
- AE:      Receita(60%) + MB%(40%) — calc_bonus_ae()
- AE_GM:   TCV Área(20%) + Receita(30%) + MB%(50%) — calc_bonus_ae() com peso_tcv=0.2
  → AE_GM são os AEs do vertical Grupo Mult: Edmilson, Guilherme, DANILO DE SOUZA MACIEIRA
  → TCV é do vertical inteiro "Grupo Mult" (não por cliente individual)
- HUNTER:  TCV(70%) + Receita(20%) + MB%(10%)
- ESTRATEGISTAS: Receita(60%) + MB%(40%)

FÓRMULA GERAL (AE/AE_GM)
-------------------------
Bônus = Σ por WS [ sal × peso_rec × peso_ws × ating_rec × lb_gate_ws
                  + sal × peso_mb  × peso_ws × ating_mb  × mb_gate × lb_gate_ws ]
       + sal × peso_tcv × ating_tcv_area   (apenas AE_GM)

WORK STREAMS (WS) — pesos padrão Q4
-------------------------------------
apps: 70%, cloud: 8%, dados: 8%, hyper: 8%, demais: 6%
Danilo (AE_GM) tem pesos customizados: apps=72%, cloud=10%, dados=0%, hyper=10%, demais=8%
(definido em parametros.xlsx/pesos_ws_pessoa)

TRIGGERS Q4
-----------
- Receita/TCV: 85% da meta → abaixo = 0, entre trigger e meta = 50%→100% linear
- MB%:         meta - 1.5pp → abaixo = 0, linear até meta
- Apps: gate MB = bloqueia bônus MB apps se ating_mb = 0

TRIGGERS Q3
-----------
- Receita/TCV: 90% da meta
- MB%:         meta - 1.0pp

GATE LB POR WS (implementado recentemente)
-------------------------------------------
- Para cada WS, verifica se LB realizado >= trigger LB proporcional
- realized_lb_gate_ws: acumula APENAS clientes com budget LB > 0 naquela WS
  (evita que clientes sem budget inflem o gate)
- trigger_lb_ws_k = bgt_lb_ws * (trigger_lb_anual / meta_lb_anual)
- Se trigger = 0 → gate auto-passa

BENCHMARKS MB% POR WS
----------------------
Cloud: 34%, Dados: 35%, Hyper: 35%, Demais: 37%
Apps: calculado de verdade (custo real ou benchmark 35% se sem custo)

RECALCULO LOCAL (FRONTEND)
---------------------------
O frontend recalcula alguns bônus localmente para exibir na tabela de WS.
Aplica lb_gate_ws ao recalcular:
  const lbGate = w.lb_gate_ws ?? 1;
  const bRec = sal * pRec * w.peso_ws * (w.ating_rec ?? 0) * lbGate;
  const gate = w.aplica_gate_mb ? (w.mb_gate ?? 1) : 1;
  const bMb  = sal * pMb  * w.peso_ws * (w.ating_mb ?? 0) * gate * lbGate;

VALORES HARDCODED / SIMPLIFICAÇÕES — IMPORTANTE PARA MANUTENÇÃO

   QUANDO TIVER DADO: adicionar linhas tipo="TCV_Q3" em realizados_manual e atualizar

2. Benchmark Apps = 35% (fallback)  (linha: marg_q4.at[_idx, "margem_pct"] = 0.35)
   MOTIVO: quando um projeto Apps não tem custo_rateado no SAP e não tem dados
   de margem_pessoas para aquele PEP+período, usa 35% de MB% como fallback.
   NÃO MUDA com os dados Q3 — é um fallback de segurança, não um dado faltante.

3. Override OpenX = 45% MB  (linhas com "openx_mask" em _load_all e _load_q3_realized)
   MOTIVO: projetos com no_hierarquia="OPENX" usam MB% fixo de 45% independente
   do SAP. Decisão de negócio, não um dado faltante.

4. _trigger_ratio para Q3 usa mesma tabela anual de parametros.xlsx/lb_trigger
   MOTIVO: essa tabela só tem valores anuais (meta_lb, trigger_lb). O ratio
   trigger/meta (~85%) é aplicado proporcionalmente ao budget Q3 de cada WS.
   Se as metas anuais mudarem para Q3, atualizar a aba lb_trigger.

5. WS_MB_BENCHMARK_Q4 usado também no Q3  (em _load_q3_realized e calc_bonus_ae_q3)
   MOTIVO: os benchmarks de MB% por WS (cloud=34%, dados=35%, etc.) são os mesmos
   para Q3 e Q4 por decisão de negócio. Se mudarem para Q3, criar WS_MB_BENCHMARK_Q3.

================================================================================
COMO ADICIONAR DADOS SAP Q3 (passo a passo)
================================================================================

Quando o usuário fornecer os dados SAP Q3, o agente deve:

PASSO 1 — operacional.xlsx / projetos
  Adicionar linhas com periodo="2025-07", "2025-08" ou "2025-09".
  Colunas obrigatórias:
    periodo, pep, nome_cliente, empresa, centro_lucro, no_hierarquia,
    categoria_bu, receita, custo_rateado, horas, tipos
  categoria_bu deve usar os valores: Apps, Cloud, Cyber, Dados, Hyper, Demais
  (o engine normaliza para minúsculas internamente via _norm_ws())
  OBS: pode usar criar_arquivos_consolidados.py se os CSVs brutos estiverem disponíveis.

PASSO 2 — operacional.xlsx / projetos (RAC)
  As linhas com RAC devem ter o campo "tipos" preenchido (ex: "MapaReceita").
  O engine explode o campo tipos e usa valor_liquido como receita RAC.

PASSO 3 — parametros.xlsx / realizados_manual (TCV Q3, se disponível)
  Adicionar linhas com tipo="TCV_Q3", ae_ou_vertical=vertical, receita=valor.
  Depois atualizar a linha hardcoded em calc_bonus_ae_q3():
    Trocar: real_tcv_q3_ae = 0.0
    Por:    real_tcv_q3_ae = float(d["tcv_real_q3"].get("Grupo Mult", 0.0))

PASSO 4 — Invalidar cache do engine (se servidor estiver rodando)
  O engine usa @lru_cache. Em produção no Render, o deploy automático reinicia
  o processo e limpa o cache. Em desenvolvimento local, reiniciar o servidor.

NÃO é necessário mexer em código para os passos 1 e 2 — o engine lê
automaticamente qualquer linha com periodo nos Q3_PERIODOS.
