SISTEMA DE ALIASES — NOMES DIFERENTES DO MESMO CLIENTE/PESSOA
================================================================================

PROBLEMA
--------
O SAP registra nomes de clientes que podem ser diferentes dos nomes usados no
budget (ex: "BANCO BMG S.A." no SAP vs "BMG" no budget).

SOLUÇÃO — campo nome_base em parametros.xlsx/clientes
------------------------------------------------------
Cada linha em "clientes" representa um cliente canônico com:
  - nome_cliente: nome exibido no dashboard (nome oficial/curto)
  - nome_base:    nome(s) como aparecem no SAP, separados por | quando múltiplos
  - bu:           vertical/BU (ex: "Grupo Mult", "Health", "Finance", "Multisector")
  - ae:           nome do AE responsável

Exemplos ativos:
  nome_cliente="SADA"     nome_base="SADA TRANSPORTES|SADA TRANSPORTES E ARMAZENAGENS LTD"
  nome_cliente="ESTAPAR"  nome_base="ESTAPAR ESTACIONAMENTOS SC LTDA|ALLPARK EMPREENDIMENTOS, PARTICIPAC"
  nome_cliente="Compass"  nome_base="COMPASS GAS E ENERGIA S.A."
  nome_cliente="Honda"    nome_base="HONDA ASSESSMENT ACESSO"

REGRA IMPORTANTE
----------------
Sempre usar o nome mais curto como nome_cliente canônico (ex: "Compass" não
"COMPASS GAS E ENERGIA S.A.", "Honda" não "HONDA ASSESSMENT ACESSO").
Nunca criar nova linha para um alias — sempre adicionar ao campo nome_base com |.
Exceção: KLABIN aparece em 2 linhas porque tem BU diferente (Multisector + Grupo Mult).

COMPORTAMENTO NO CÓDIGO
-----------------------
- main.py: _clientes_lookup() constrói ae_map e vertical_map a partir dos aliases
- apuracao_engine.py: usa nome_base para normalizar nomes SAP
  → IMPORTANTE: nome_base é pipe-separated, o código FAZ split por "|" para criar
    uma entrada por alias. Não tratar o nome_base inteiro como chave única.
  → bloco _alias_pre: for _alias in [a.strip() for a in _nb.split("|") if a.strip()]:
        _alias_pre[norm(_alias)] = _nc_n
  → mesmo padrão no segundo bloco de alias no loop principal

================================================================================
REGRAS DE NEGÓCIO — BÔNUS Q4 2025
================================================================================

POSIÇÕES
--------
- DIRETOR: TCV(20%) + Receita(30%) + MB%(10%) + MC%(40%) — calc_bonus_diretor()
- AE:      Receita(60%) + MB%(40%) — calc_bonus_ae()
- AE_GM:   TCV Área(20%) + Receita(30%) + MB%(50%) — calc_bonus_ae() com peso_tcv=0.2
  → AE_GM são os AEs do vertical Grupo Mult: Edmilson, Guilherme, DANILO DE SOUZA MACIEIRA
  → TCV é do vertical inteiro "Grupo Mult" (não por cliente individual)
- HUNTER:  TCV(70%) + Receita(20%) + MB%(10%)
- ESTRATEGISTAS: Receita(60%) + MB%(40%)

FÓRMULA GERAL (AE/AE_GM)
-------------------------
Bônus por WS = sal × peso_rec × peso_ws × ating_rec           ← bônus receita
             + sal × peso_mb  × peso_ws × ating_mb × mb_gate × lb_gate_ws × rec_gate

ATENÇÃO: lb_gate_ws NÃO entra no bônus de receita (só no MB).
rec_gate = 1 se ating_rec > 0, senão 0 (MB bloqueado se receita zerada)

WORK STREAMS (WS) — pesos padrão Q4
-------------------------------------
apps: 70%, cloud: 8%, dados: 8%, hyper: 8%, demais: 6%
Fabio (FABIO LUIS BERTI BARCELOS) pesos customizados:
  cloud=10%, apps=72%, dados=10%, hyper=10%, demais=0%
  → demais peso=0 → nunca gera bônus demais para Fabio

TRIGGERS Q4
-----------
- Receita/TCV: 85% da meta → abaixo = 0, entre trigger e meta = 50%→100% linear
- MB%:         meta - 1.5pp → abaixo = 0, linear até meta
- Apps: gate MB = bloqueia bônus MB apps se ating_mb = 0

ATINGIMENTO PARCIAL
-------------------
calc_atingimento(realizado, meta, trigger=0.85):
  < meta * trigger  → 0.0
  >= meta           → 1.0
  entre ambos       → (realizado - meta*trigger) / (meta - meta*trigger) * 0.5 + 0.5
  Retorna 0.5 a 1.0 para resultados entre trigger e meta (linear)

calc_atingimento_mb(real_pct, meta_pct, delta=0.015):
  trigger_abs = meta_pct - delta
  Mesma lógica: 0 abaixo, 0.5→1.0 linear, 1.0 acima

TRIGGERS Q3
-----------
- Receita/TCV: 90% da meta
- MB%:         meta - 1.0pp

GATE LB POR WS
--------------
- Para cada WS, verifica se LB realizado >= trigger LB proporcional
- realized_lb_gate_ws: acumula APENAS clientes com budget LB > 0 naquela WS
- trigger_lb_ws_k = bgt_lb_ws * _trigger_ratio
- Se trigger = 0 → gate auto-passa (lb_gate_ws = 1)

CÁLCULO DE RECEITA REAL POR CLIENTE
--------------------------------------
real_rec = max(
    _match_cliente(cli_n, rac_by_client_nh),    ← receita RAC (tipos != "")
    _match_cliente(cli_n, rec_by_client_nh),    ← receita projetos/SAP
)
O max() garante que clientes com RAC parcial (ex: Compass com R$30k RAC)
não percam a receita total que está em rec_by_client_nh.

Clientes sem RAC (tipos="") como Honda e alguns da Compass: real_rec vem de
rec_by_client_nh (projetos SAP).

CAMPOS RETORNADOS POR calc_bonus_ae()

budget_mb_pct, real_mb_pct, ating_mb_total
meta_lb_q4, trigger_lb_q4, real_lb_total, lb_gate
bonus_total
detalhe_ws: lista por WS (ws, peso_ws, budget_rec, real_rec, ating_rec,
  budget_mb_pct, real_mb_pct, ating_mb, mb_gate, aplica_gate_mb,
  meta_lb_ws, trigger_lb_ws, real_lb_financeiro, real_lb_gate, lb_gate_ws,
  bonus_rec, bonus_mb, bonus_ws, clientes_ws)
